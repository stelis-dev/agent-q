# Agent-Q Implementation Status

This document is the source of truth for what Agent-Q currently implements
across the common protocol, host process, firmware targets, chain adapters, and
security profiles.

Legend:

- `O`: implemented and verified for at least one target.
- `△`: partially implemented, designed but not complete, or implemented only as
  a local diagnostic/self-test.
- `X`: not implemented.
- `N/A`: not applicable to that target.

Sign API activation uses these status terms:

- `foundation`: standalone source modules, protocol shapes, host tests, and
  target-internal partial runtime modules may exist, but no public Firmware USB
  dispatcher, public response writer, host process/client/provider API, parser,
  capability advertisement, or MCP surface is active. If internal review, PIN,
  history, signing, or cleanup modules exist, the relevant row must say so
  explicitly; this status is still not product exposure.
- `source-wired-not-product-active`: public dispatcher, response writer,
  host process/client/MCP/provider parser and API, capability advertisement, and
  target runtime glue are present in source for the current contract, but
  current-tree product-active evidence is not yet complete. This status means
  the source has public exposure and must not be called `product-active`.
- `product-active`: public dispatcher, response writer, client/provider parser
  and API, capability advertisement, docs, tests, build, and current-tree target
  hardware and visual evidence are all complete for the same contract.

Current `sign_transaction`, `sign_personal_message`, and Sui zkLogin active
identity support statuses are `source-wired-not-product-active`. The signing
wire requests are public in source. Firmware chooses the `sign_transaction`
authorization gate from the device-local signing authorization mode, while
`sign_personal_message` is user-mode only and fails closed in policy mode. Sui
`sign_transaction` can receive sender-bound sponsored transactions only when the
active account's device-local Sui account setting accepts gas sponsors; it still
returns only the active sender signature. Sui zkLogin preparation/proposal is
exposed through common `credential_prepare` and `credential_propose` protocol
operations, and final native-vs-zkLogin signature envelope construction happens
only after the existing signing authorization gate. The Sign API and zkLogin
credential path are source-wired but not
`product-active`: the tracked status does not yet contain complete
source/docs/tests/build/hardware/visual evidence for the same contract.

The current shared signing boundary classifies bounded `(type, chain, method)`
routes explicitly in Core, host process, and Firmware. Sui is the only executable
chain; syntactically valid unsupported chains fail as `unsupported_chain`, and
unsupported or type-mismatched Sui methods fail as `unsupported_method`.
Common core, host process, and MCP request validation own transport-envelope
bounds and canonical base64 syntax. Sui signing decoded-byte capacities are part
of the shared host/Firmware signing contract and are enforced before request
delivery in Core and again inside Firmware. Public capabilities advertise
signing availability and read-only Firmware signing mode; they do not expose
direct/upload transport thresholds for adapters to choose.

The modal UI uses a NES light theme defined by one color/shape token table
(`ui_theme.h`), pins display brightness to 50, and routes
generate/import Cancel back to the setup-choice menu. On-device rendering and
touch for the import, error, and reset modals are not yet hardware-verified.

No `product-active` claim holds until the tracked status row says complete
source/docs/tests/build/hardware/visual evidence exists for that contract.

This document tracks implementation status only. The wire protocol is defined in
`specs/PROTOCOL.md`. Target-specific details live under
`firmware/src/<hardware-id>/`.

| Buffered result recovery (`get_result` / `ack_result`) | △ | StackChan CoreS3 source buffers each completed signing response in bounded RAM under `(session, request id)` together with a form-discriminated internal identity of the selected route and validated signing parameters. Re-submitting the same id replays only when that identity matches and the entry is still retained; conflicting reuse fails with `request_id_conflict` before adapter or authorization work only while the original entry remains buffered. `get_result` retrieves an already-stored response by id (or `unknown_request` when none is buffered), and `ack_result` releases it; those recovery operations do not resubmit a signing request. The shared host core USB signing path and browser provider attempt recovery only for signing transport failures after the write side effect for the original request has started; open, flush, port acquisition, and queued-deadline failures remain non-recoverable. After a buffered result is recovered, `ack_result` runs inside the same signing-recovery transaction so later host/provider requests cannot overtake cleanup; `ack_result` failure still does not turn a recovered signing outcome into a failure. A Firmware-observed `invalid_session` during `get_result` ends the host/provider session view; `ack_result invalid_session` keeps the recovered result visible but still clears the host/provider session mirror. The host core and browser provider transport paths use port-lease guards: deadline expiry prevents later code from starting a new write side effect, already-started writes are treated as recovery candidates, the Node path blocks the same `portPath` while stale cleanup is pending, and the browser provider blocks the same `SerialPort` object unless cleanup is proven; if cleanup cannot be proven, later requests fail or reacquire a different port instead of reusing the stale object. Firmware classifies `get_result` and `ack_result` as retained-response read/cleanup routes, not signing ingress, so they are silent busy-gate exceptions but still require provisioned material and an active matching session. The Firmware buffer is session-bound, currently fixed-size with oldest-entry eviction, can survive a transient USB link drop through a short Firmware session grace window, and is cleared on ack, session end, on a new session replacing it, wipe, or device reset. The browser provider can recover within the active signing transaction after a physical disconnect/reselection when Firmware still accepts the captured session. The Node host core recovery path is same-`portPath` automatic recovery; it does not claim automatic cross-`portPath` recovery after USB re-enumeration. Host firmware tests cover identity match/conflict, request-id-conflict retry-delivery mapping, response-store ack release, capacity eviction, session cleanup, grace decision logic, request-line framing, and retained-response route classification; host core USB tests and browser provider tests cover `get_result` recovery, `get_result` / `ack_result` `invalid_session` session-mirror effects, original-error preservation when recovery fails, non-recovery before write start, port-lease timeout containment, transaction ordering, same-port stale reuse blocking, post-write disconnect recovery, and the `ack_result` round-trip. Live USB link behavior hardware evidence follows the Sign API evidence paragraph above. |

## Explicitly Unsupported Or Out Of Scope

- Ledger-grade or secure-element-backed physical extraction resistance.
- Agent, prompt, or host intent detection.
- Chain-specific top-level MCP tools such as `sign_sui_transaction`.
- the host process, MCP, or Admin commands that export keys, directly set or clear
  policy, flash firmware, read memory, or bypass Firmware policy.
- Fiat cash-out, P&L, tax, or cost-basis features.
- Treating client labels or purpose routing as security policy.
- Treating `connect_device` as signing approval.
